### 선수 지식이 존재하는가?

AWS를 사용하여 프로젝트를 배포하기 위해서는 리눅스와 네트워크에 대한 지식이 필요합니다.

### Iaas 와 Paas

프로젝트를 간단하게 배포하기 위해 EC2 환경을 구축할 수 있습니다. EC2는 IaaS로 분류되며, 물리적인 컴퓨터를 가상화하여 제공합니다. 이를 코드로 구축하거나 Shell 스크립트를 사용하여 효율적으로 관리할 수 있습니다.

또한, 환경 구축 없이 프로젝트를 배포하는 방법으로 Elastic Beanstalk을 활용할 수 있습니다. Elastic Beanstalk는 플랫폼-어즈-어-서비스(PaaS)로 분류되며, 미리 구축된 환경을 제공합니다. 예를 들어, 자바 프로젝트를 배포한다면 JDK가 이미 설치된 환경을 사용할 수 있습니다.

### Github Actions

배포 자동화를 위해 Github Actions를 사용할 수 있습니다. 이는 Github에서 제공하는 기능으로, 코드를 Github에 푸시하면 트리거가 발생하여 지정된 작업을 자동으로 수행합니다.

### 무중단 배포

무중단 배포를 구현하기 위해서는 로드 밸런싱과 롤링 배포에 대한 이해가 필요합니다. Nginx를 직접 구축하여 사용할 수도 있습니다. 로드 밸런서를 통해 트래픽을 여러 EC2 서버로 분산시킨 후, 롤링 배포를 수행합니다. 롤링 배포는 새로운 버전의 서비스를 점진적으로 배포하는 방식입니다.

### 정적 IP 할당

정적 IP 할당을 위해 Network Load Balancer를 사용할 수 있습니다. Network Load Balancer는 고정 IP 주소를 제공하여 서비스에 연결할 수 있도록 도와줍니다.

### 진행 방향

진행 방향으로는 스프링 부트를 사용하여 간단한 프로젝트를 만들고, 테스트 주도 개발(TDD)을 적용하여 테스트 코드를 작성합니다. 이를 Github에 업로드하면 Github Actions에서 트리거가 발생합니다. 트리거가 발생하면 다음 작업이 자동으로 수행됩니다:

1. 테스트 실행 및 확인
2. 빌드 (jar 파일 생성) 및 확인
3. 빌드가 완료되면 프로젝트가 AWS로 자동 배포됩니다

AWS의 가장 앞단에는 로드 밸런서를 위치시킵니다. 로드 밸런서는 트래픽을 여러 EC2 서버로 분산 시킵니다. 로드 밸런서는 EC2와 연결되어 있습니다.

### 로드 밸런서

로드 밸런서는 배포 시 새로운 EC2 서버를 기존의 서버와 병렬로 복제합니다. 이렇게 복제된 서버들은 새로운 버전의 애플리케이션을 호스팅합니다. 배포 전에는 기존의 서버가 트래픽을 처리하고 있으므로 안정성을 확인할 수 있습니다.

새로운 버전의 서버가 정상적으로 작동하는지 확인한 후, 롤링 배포를 진행합니다. 이때, 로드 밸런서는 트래픽을 점진적으로 새로운 서버로 전달합니다. 예를 들어, 처음에는 10%의 트래픽을 새로운 서버로 보내고, 이후 점차적으로 트래픽의 비중을 증가시킵니다.

롤링 배포를 통해 새로운 버전의 서비스가 안정적으로 작동하는지 확인하고, 문제가 발생하지 않는다면 기존의 서버를 하나씩 제거하여 전환을 완료합니다. 이렇게 하면 서비스 중단 없이 새로운 버전으로 업데이트할 수 있습니다.

### 온디맨드

클라이언트 수가 증가하면 자동으로 EC2 서버를 확장할 수 있습니다. 이는 AWS의 **온 디맨드** 기능으로, 사용자 수가 증가하면 자동으로 서버 수를 늘리고, 사용자 수가 감소하면 자동으로 서버 수를 줄입니다. 이를 통해 자원을 효율적으로 관리하면서 사용자에게 일관된 서비스 품질을 제공할 수 있습니다.
