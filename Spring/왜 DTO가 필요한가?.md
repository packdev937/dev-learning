# 왜 DTO가 필요한가

### 개요

서비스가 요청을 처리하고 클라인트로 반환할 때, 엔티티 그 자체를 반환하는 것이 아닌 데이터를 전달하기 위해 사용하는 오브젝트인 Data Transfer Object를 이용합니다. 왜 굳이 Model을 놔두고 DTO를 새로 생성해서 반환할까요?

### 캡슐화

첫 번째로 비즈니스 로직을 캡슐화 하기 위해서 입니다. 대게 엔티티는 데이터베이스 테이블 구조와 매우 유사합니다. JPA를 활용하는 로직을 봐도 @Entity 매핑 하나면 필드 값들을 각각 데이터 베이스의 컬럼에 자동으로 매핑해주죠. 하지만 외부인이 데이터베이스의 스키마를 파악하는 것은 누구도 원치 않는 일입니다. 이 때 DTO와 같이 다른 오브젝트로 필요한 필드 값만 넣어서 반환해주면 서비스 내부 로직 혹은 데이터베이스 구조를 숨길 수 있습니다.

### 예외 케이스

클라이언트가 요청하는 정보를 엔티티가 모두 포함하고 있지는 않습니다. 예를 들어 에러 메세지의 경우 이를 엔티티에 담기는 매우 애매합니다. 이 경우 별도의 DTO를 만들어서 에러 메세지를 포함하면 됩니다.

### API 스펙

서비스를 개발할 때 엔티티는 변해도 API 스펙이 변경되면 안됩니다. 가령 초기의 API 스펙은 회원의 이름, 이메일만 반환하기로 했지만, 엔티티가 성별을 추가하므로써 API 스펙도 성별을 반환하면 안된다는 얘기입니다. 하지만 엔티티로 반환하게 되면 이를 방지하기 매우 어렵습니다. 따라서 별도의 API 스펙을 DTO로서 정의하고 해당 DTO를 반환하는 것이 바람직합니다.

### 응용

회원가입을 한다고 가정해봅시다. 이 때는 DTO로 매핑된 객체를 Entity로 변환합니다. 반대로 클라이언트가 리스트를 조회했을 때는 service에서 불러온 Entity 리스트를 DTO로 변환해줍니다. 둘의 공통점은 외부에서 넘어온 혹은 넘어가는 객체들은 모두 DTO 인스턴스라는 점입니다.
